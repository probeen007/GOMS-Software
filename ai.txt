# Build Prompt: Garage / Auto-Service Management System (MERN MVP)
### Stepwise, Module-by-Module Build Plan

Use this entire document as the prompt for the AI coding tool (e.g. Claude Code).

## 📋 Context — What We Are Building

We are building a **Garage Management System**: a web application used by an auto-repair shop (garage) to run its entire day-to-day operation digitally, replacing paper job cards, manual invoices, and spreadsheet-based inventory tracking.

**Who uses it:** Garage staff only (admin, receptionist, technician, accountant) log in and use it daily. Customers don't get accounts — their only touchpoint is a public, no-login quotation-approval page reached via a shared link.

**The core journey it manages, end to end:**
A customer calls or walks in → receptionist books an **appointment** → vehicle arrives and is **checked in** (mileage, condition photos) → technician inspects it and a **quotation** is built → customer approves the quotation → a **job card** is opened automatically and the technician does the work, pulling parts from inventory → the job card is closed → an **invoice** is generated automatically → the customer pays and the payment is recorded → **loyalty points** are credited → all of this rolls up into **finance reports**, **inventory stock levels**, **notifications**, and an admin **dashboard**.

**Why it matters:** every stage automatically triggers the next where possible (approve quote → job card opens; close job card → invoice generates; full payment → loyalty points credited; purchase recorded → expense logged), so staff don't have to manually duplicate data entry across modules. That automation is the core value of the system — get it working correctly.

**What this MVP intentionally is NOT:** there are no real SMS/WhatsApp/email integrations and no real payment gateway. Notifications are shown in-app, and payments are recorded manually by staff. Everything else in the journey above is real, working functionality — not a mockup.

**How it will be built:** as a MERN (MongoDB, Express, React, Node) app, with a clean, professional, fully responsive UI, delivered one module at a time as described below.

## ⚠️ How to build this — read before starting

- Build **one module at a time, in the exact order listed below.**
- A module is only considered "done" when it is **fully functional end-to-end**: backend (model + routes + controller logic) AND frontend (working UI connected to the API) AND it has been verified to run without errors.
- Do **not** move to the next module until the current one is complete.
- After finishing each module, **update this document**: change its checkbox from `[ ]` to `[x]`, and add a one-line note under it stating what was built and confirming it works (e.g. `✅ Done — CRUD tested, responsive UI verified`).
- Before starting a new module, **re-read the checklist** in Section 0 to confirm what's already done and what's next, so progress is never lost or duplicated.
- At the end of every module, print a short progress summary: how many modules complete out of total, and which module is next.
- If a later module depends on an earlier one (e.g. Job Cards depend on Quotations), do not stub it out — the dependency must already be real and working because it was built in an earlier step.

## Tech Stack (exact choices — use for every module)

- **Frontend:** React (Vite), React Router v6, Axios, Context API or Redux Toolkit, Tailwind CSS
- **Backend:** Node.js + Express.js
- **Database:** MongoDB with Mongoose ODM
- **Auth:** JWT (jsonwebtoken) + bcrypt
- **PDF generation:** Puppeteer
- **Scheduled jobs:** node-cron
- **File uploads:** Multer, local disk storage under `/uploads`
- **Validation:** express-validator (backend) + form validation (frontend)
- **Charts:** Recharts

Constraint for the whole app: every feature must be achievable with code alone. No real SMS, WhatsApp, email, or payment-gateway integrations anywhere. Notifications are in-app; payments are recorded manually by staff.

## Reference: Database Schema (Mongoose Models)

Add `createdAt`/`updatedAt` timestamps to all collections.

- **User**: name, email, passwordHash, role, isActive
- **Customer**: name, phone, email, address, type, loyaltyPoints, deletedAt
- **Vehicle**: customerId (ref), plateNo, make, model, year, vin, colour
- **Appointment**: customerId, vehicleId, staffId (technician), apptDate, serviceType, status, source, notes, deletedAt
- **CheckIn**: appointmentId, mileageIn, conditionNotes, photoUrls[]
- **Quotation**: appointmentId, customerId, approvalToken, status, validUntil, lineItems[{type, description, qty, unitPrice, total}], subtotal, discount, vat, total
- **JobCard**: appointmentId, quotationId, vehicleId, technicianId, mileageIn, mileageOut, complaint, findings, status, partsUsed[{partId, qty, costAtTime}], labourEntries[{description, amount}], labourCost, partsCost, openedAt, closedAt
- **Invoice**: jobCardId, customerId, invoiceNo, subtotal, discount, vat, total, status, pdfPath, issuedAt
- **Payment**: invoiceId, amount, method, referenceNote, paidAt
- **InventoryStock**: sku, name, supplierName, qty, minQty, unitCost, unitPrice
- **Purchase**: supplierName, items[{partId, qty, unitCost}], totalCost, billFileUrl, createdAt
- **Expenditure**: category, amount, note, date
- **LoyaltyLedger**: customerId, change (+/-), reason, refType, refId
- **Notification**: recipientRole or recipientUserId, message, type, isRead, createdAt
- **AuditLog**: userId, action, collection, recordId, diff, timestamp
